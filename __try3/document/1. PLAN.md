# `plan.md` 작성 계획: CCTV 교통사고 검출 정확도 향상 고려사항 정리

## Summary
`plan.md`는 현재 [`qwen_bone.py`](/root/Desktop/workspace/ja/qwen_bone.py) 기준의 파이프라인을 출발점으로 삼되, 문서 성격은 "연구 가이드"로 둔다. 문서는 코드 설명서가 아니라, 정확도를 높이기 위해 어떤 축을 점검해야 하는지와 왜 그것이 중요한지를 구조적으로 설명하는 문서로 작성한다.

핵심 서술 흐름은 다음으로 고정한다:
1. 프로젝트 목표와 현재 파이프라인 요약
2. 정확도 평가 기준 정의
3. 정확도에 영향을 주는 핵심 요인들
4. 현재 구현의 한계와 그로 인한 오류 유형
5. 우선순위가 있는 개선 방향
6. 실험/검증 체크리스트

문서는 한국어로 작성하고, 예시는 현재 프로젝트의 3개 예측 대상인 `accident_time`, `center_x/center_y`, `type`에 맞춘다.

## Public APIs / Interfaces / Types
코드 API 변경은 없다. 다만 `plan.md` 안에서는 현재 예측 인터페이스를 명시적으로 설명한다.

반드시 문서에 포함할 인터페이스 설명:
- 입력: CCTV 영상 + 메타데이터(`region`, `scene_layout`, `weather`, `day_time`, `quality`, `duration`, `no_frames`, `height`, `width`)
- 출력:
  - `accident_time`: 사고 시작 시점(초)
  - `center_x`, `center_y`: 충돌 중심의 정규화 좌표
  - `type`: `rear-end | head-on | sideswipe | t-bone | single`
- 현재 파이프라인:
  - 1단계: 비디오 전체에서 사고 시점 탐지
  - 2단계: 해당 시점 프레임 추출
  - 3단계: 단일 프레임으로 위치 추정
  - 4단계: 단일 프레임으로 사고 유형 분류

## `plan.md` 문서 구조
다음 섹션과 핵심 내용을 그대로 담는 방향으로 작성한다.

### 1. 목표 정의
- 이 프로젝트의 목적은 CCTV 영상에서 교통사고의 "정확한 시점", "정확한 위치", "정확한 사고 타입"을 검출하는 것이라고 명시한다.
- 세 과제가 서로 의존적이라는 점을 설명한다.
- 특히 시점 오차가 위치/타입 오차를 연쇄적으로 키운다는 점을 강조한다.

### 2. 현재 접근 방식과 기본 한계
[`qwen_bone.py`](/root/Desktop/workspace/ja/qwen_bone.py)를 바탕으로 현재 구조를 짧게 설명한다.
- 시간 탐지는 비디오 입력 기반 VLM 추론
- 위치와 타입은 사고 시점의 단일 프레임 기반 추론
- JSON 강제 출력, 단순 유효성 검증, 재시도 로직 사용

반드시 현재 한계로 적을 항목:
- `VIDEO_FPS = 1.0` 기반 샘플링은 정확한 사고 시작 시점을 놓치기 쉽다.
- 위치/타입을 단일 프레임 하나에 의존하면 접촉 직전/직후 문맥을 잃는다.
- 시간 예측 실패를 단순 재시도로 처리해도 근본적 근거가 강화되지는 않는다.
- 현재 검증 로직은 형식 검증 중심이고 의미 검증은 약하다.
- 메타데이터는 활용하지만, 오탐/편향을 막는 체계적 사용 원칙은 없다.

### 3. 정확도 평가 기준을 먼저 명확히 해야 하는 이유
이 섹션은 연구 가이드의 중심이다. 다음 지표들을 구분해서 설명한다.
- 시점 정확도:
  - 절대 오차(초)
  - 허용 오차 내 적중률
  - coarse-to-fine 평가 필요성
- 위치 정확도:
  - 정답 위치와의 픽셀 거리 또는 정규화 거리
  - 허용 반경 내 적중률
  - 가능하면 bbox/충돌영역 annotation 필요
- 타입 정확도:
  - Accuracy
  - 클래스별 Precision/Recall/F1
  - Confusion matrix
- 종합 성능:
  - 세 항목 모두 일정 기준을 만족해야 성공으로 보는 joint metric 필요

이 섹션에서는 "무엇을 높일 것인지 정의되지 않으면 개선이 아니라 튜닝만 하게 된다"는 메시지를 분명히 둔다.

### 4. 데이터와 라벨 품질에서 봐야 할 점
정확도 향상에서 가장 큰 영향 요인으로 데이터 품질을 둔다.
반드시 포함할 세부 항목:
- 사고 시작 시점 라벨 기준의 일관성
- 위치 라벨의 정의 일관성
  - 차량 중심이 아니라 실제 충돌 지점 중심인지
- 타입 라벨 경계의 명확성
  - `rear-end` vs `sideswipe`
  - `t-bone` vs 다중 차량 복합 충돌
- CCTV 품질 편차
  - 야간, 우천, 저해상도, 압축 손상, 원거리 카메라
- 클래스 불균형
  - 특정 타입만 많으면 타입 분류가 왜곡됨
- 장면 편향
  - 특정 교차로 구조나 국가/지역 특성에 과적합될 수 있음

여기서는 "모델을 바꾸기 전에 정답 정의가 흔들리지 않는지 확인해야 한다"는 점을 강조한다.

### 5. 시간 검출 정확도를 높일 때 고려할 점
이 섹션은 현재 구현의 가장 큰 병목으로 정리한다.
반드시 넣을 포인트:
- 1fps 샘플링은 충돌 순간 포착에 너무 거칠다.
- coarse-to-fine 탐색이 필요하다.
  - 1차: 저해상도/저fps로 후보 구간 탐색
  - 2차: 후보 구간만 고fps 재분석
- "충돌 발생 프레임"과 "회피 불가능해진 프레임"을 구분해 라벨 기준을 통일해야 한다.
- 비디오 전체를 한 번에 판단시키기보다 구간별 reasoning 없이 후보 비교 구조가 더 안정적일 수 있다.
- optical flow, trajectory, object tracking 같은 시간 정보가 중요하다.
- 사고 직전 몇 초의 감속, 접근 속도, 상대 각도 같은 동역학 특징이 도움이 된다.
- 단순 재시도보다 self-consistency나 다중 후보 산출 후 검증이 더 낫다.

현재 코드와 연결해서 언급할 점:
- [`qwen_bone.py`](/root/Desktop/workspace/ja/qwen_bone.py)의 `MAX_TIME_RETRIES`, `TIME_MATCH_THRESHOLD`는 출력 안정화 장치일 뿐, 근본적인 시간 분해능 문제를 해결하지 못한다.

### 6. 위치 검출 정확도를 높일 때 고려할 점
반드시 포함할 포인트:
- 단일 프레임만으로는 실제 충돌 중심을 놓칠 수 있다.
- 사고 전후 인접 프레임을 함께 봐야 접촉점이 선명해진다.
- 전체 프레임에서 바로 좌표를 찍게 하기보다 사고 관련 ROI를 먼저 좁히는 것이 유리하다.
- 작은 차량, 원거리 충돌, 가림(occlusion), 야간 헤드라이트가 위치 오차를 키운다.
- tracking 결과나 motion hotspot을 함께 쓰면 충돌 지점 추정이 안정적이다.
- 좌표값은 형식적으로 0~1 범위여도 의미상 틀릴 수 있으므로 시각적 검증 체계가 필요하다.

현재 코드와 연결해서 언급할 점:
- 현재는 사고 시점 프레임 1장만 저장해 위치를 추정하므로, 정확도 향상을 위해서는 다중 프레임, crop, detection/tracking 보조정보 사용을 검토해야 한다.

### 7. 사고 타입 분류 정확도를 높일 때 고려할 점
반드시 포함할 포인트:
- 타입 분류는 정적인 모양보다 충돌 직전/직후 상대 운동 방향이 더 중요하다.
- `rear-end`, `head-on`, `sideswipe`, `t-bone`은 관계형 분류이므로 단일 이미지보다 짧은 시퀀스가 유리하다.
- 복수 차량이 동시에 등장하는 경우 "주 사고"를 어떤 기준으로 정할지 필요하다.
- 카메라 각도에 따라 `rear-end`와 `sideswipe`, `t-bone`이 혼동될 수 있다.
- 타입 불확실성을 허용하는 보조 점수나 후보 순위도 실험 가치가 있다.
- confusion matrix 기반 분석이 필수다.

현재 코드와 연결해서 언급할 점:
- 현재 [`qwen_bone.py`](/root/Desktop/workspace/ja/qwen_bone.py)는 사고 시점 프레임 1장으로 타입을 분류하므로, 방향성 정보 손실이 크다고 명시한다.

### 8. 프롬프트와 출력 설계에서 고려할 점
연구 가이드지만 프롬프트 영향은 별도 섹션으로 둔다.
반드시 포함할 항목:
- 과업을 한 번에 많이 시키지 말고 분리하되, 단계 간 오류 전파를 관리해야 함
- 출력 형식 강제는 필요하지만, 형식 안정성이 의미 정확도를 보장하지는 않음
- 정의가 애매한 타입은 프롬프트 안에서 경계 사례까지 명시해야 함
- "첫 충돌 시점"의 정의를 더 엄격히 적어야 함
- few-shot 예시나 hard negative 예시가 성능에 도움이 될 수 있음
- 프롬프트 수정은 반드시 동일한 평가셋으로 비교해야 함

### 9. 전처리와 보조 신호에서 고려할 점
반드시 포함할 포인트:
- 해상도 향상, 밝기 보정, 노이즈 제거, stabilization
- 객체 검출/추적 기반 후보 영역 추출
- optical flow, trajectory, TTC 유사 특징
- 장면 메타데이터 활용
- 단, 메타데이터는 편향을 강화할 수 있으므로 "보조 힌트"로만 써야 함

여기서 [`_try1/qwen_all_inference_ver5.py`](/root/Desktop/workspace/ja/_try1/qwen_all_inference_ver5.py)가 trajectory/feature 기반 확장 방향을 시도하고 있다는 점을 간단히 연결해도 된다.

### 10. 오류 분석 체계를 반드시 만들어야 하는 이유
이 섹션에서는 단순 평균 점수보다 failure taxonomy를 강조한다.
오류 분류 예시:
- 사고 시점을 너무 이르게 예측
- 사고 시점을 너무 늦게 예측
- 비사고 이벤트를 사고로 오탐
- 충돌 차량은 맞지만 위치 중심이 틀림
- 타입을 시각적으로 비슷한 클래스로 혼동
- 야간/우천/저화질 조건에서만 반복 실패

문서에는 "성능 향상 실험은 항상 오류 유형별로 기록해야 한다"는 원칙을 넣는다.

### 11. 우선순위가 높은 개선 방향
연구 가이드이지만 우선순위는 명확히 적는다.
우선순위는 다음 순서로 고정한다:
1. 평가셋과 세부 지표 정립
2. 시간 검출의 coarse-to-fine화
3. 위치/타입의 단일 프레임 의존 완화
4. 오류 분석 체계 구축
5. tracking/trajectory 등 보조 신호 결합
6. 프롬프트 정교화
7. 필요 시 더 큰 모델, 앙상블, 미세조정 검토

각 항목마다 "왜 먼저 해야 하는지" 한 줄 설명을 붙인다.

### 12. 실험 체크리스트
문서 끝에는 바로 쓸 수 있는 체크리스트를 둔다.
체크리스트 예시:
- 동일 평가셋으로 비교했는가
- 시점/위치/타입을 분리해서 개선 효과를 측정했는가
- joint metric도 같이 봤는가
- 야간/우천/저화질 subset 성능을 따로 봤는가
- 클래스별 confusion matrix를 확인했는가
- 실패 사례를 저장하고 시각적으로 검토했는가
- 추론 비용 증가 대비 성능 향상폭을 기록했는가

## Test Cases / Scenarios
문서 자체에 포함할 검증 시나리오는 다음으로 고정한다.
- 주간, 고화질, 차량 2대가 명확히 보이는 쉬운 rear-end 사례
- 야간, 저조도, 헤드라이트 반사로 위치가 흐려지는 사례
- 교차로 장면에서 `t-bone`과 `sideswipe`가 혼동되는 사례
- 단일 차량 충돌인데 주변 차량 때문에 multi-vehicle처럼 보이는 사례

- 가림이나 프레임 밖 진입으로 충돌 지점이 일부만 보이는 사례

이 시나리오들은 "정확도 향상 포인트를 검증하는 대표 상황"으로 문서에 포함한다.
