# `qwen_frame_vllm.py` 상세 분석

## 1. 이 파일이 하는 일

`qwen_frame_vllm.py`는 CCTV 사고 영상 하나를 입력으로 받아 아래 3개 값을 추론하는 전체 실행 스크립트다.

- `accident_time`
- `center_x`, `center_y`
- `type`

겉으로 보면 "비디오를 넣고 답을 뽑는 코드"처럼 보이지만, 실제 구조는 조금 더 분리되어 있다.

1. 먼저 비디오 전체를 저해상도 프레임 시퀀스로 샘플링해 **사고가 시작된 프레임 인덱스**를 찾는다.
2. 그 인덱스를 실제 시간(`accident_time`)으로 바꾼다.
3. 그 시간의 **실제 단일 프레임**을 원본 비디오에서 다시 추출한다.
4. 그 단일 프레임으로 위치(`center_x`, `center_y`)와 유형(`type`)을 따로 추론한다.

즉 이 파일의 핵심은 단순한 "영상 전체 직접 추론"이 아니라,

- 시간은 `샘플 프레임 시퀀스` 기반으로 찾고
- 위치/유형은 `원본에서 다시 뽑은 1장 프레임` 기반으로 찾는

**2단 추론 파이프라인**이라는 점이다.

---

## 2. 전체 파이프라인 한눈에 보기

코드 흐름은 `main()` 기준으로 보면 아래 순서다.

1. 메타데이터 CSV 로드
2. 비디오 파일명 매칭
3. vLLM 엔진 로드
4. 각 영상에 대해
5. 비디오를 샘플링해서 time stage 실행
6. 선택된 시점의 실제 프레임 추출
7. 그 프레임으로 location stage 실행
8. 같은 프레임으로 type stage 실행
9. `submission.csv`와 `raw_outputs.jsonl` 저장

실행의 중심은 `main()`의 4단계다.

- time stage: `qwen_frame_vllm.py:1039-1101`
- frame extraction: `qwen_frame_vllm.py:1103-1114`
- location stage: `qwen_frame_vllm.py:1116-1127`
- type stage: `qwen_frame_vllm.py:1129-1139`

이 이름 때문에 파일명이 `frame`인 이유도 이해할 수 있다. 시간 추론은 여러 프레임을 보지만, 최종 위치와 유형은 결국 **핵심 프레임 1장**에 걸려 있다.

## 3. 핵심 설정값이 의미하는 것

초반 상수 정의는 단순한 환경 설정이 아니라 추론 성격을 결정한다.

### 3.1 모델과 생성 설정

`qwen_frame_vllm.py:21-35`

- 모델은 `Qwen/Qwen3.5-9B`
- `TEMPERATURE = 0.0`
- `TOP_P = 1.0`
- `REPETITION_PENALTY = 1.2`
- `MAX_NEW_TOKENS = 15360`

이 조합은 창의적인 생성보다 **결정론적 JSON 응답**에 맞춘 세팅이다.  
특히 `TEMPERATURE = 0.0`은 사고 시점, 위치, 유형처럼 정답형 추론에서 흔들림을 줄이려는 의도가 강하다.

### 3.2 프레임 샘플링과 멀티모달 예산

`qwen_frame_vllm.py:42-61`

- `VIDEO_FPS = 2.0`
- `MAX_VIDEO_SAMPLED_FRAMES = 64`
- `TIME_STAGE_MAX_PIXELS = MIN_PIXELS`
- `MIN_PIXELS = 256 * 28 * 28`
- `MAX_PIXELS = 1024 * 28 * 28`

여기서 중요한 건 time stage가 원본 영상을 그대로 넣지 않는다는 점이다.

- 초당 2프레임 정도만 샘플링하고
- 최대 64장으로 줄이고
- time stage에서는 더 작은 픽셀 예산을 사용한다

즉, time stage는 **정밀 판독 단계**가 아니라 **후보 시점 탐색 단계**다.

### 3.3 재시도 설정

`qwen_frame_vllm.py:56-61`

- `MAX_TIME_RETRIES = 2`
- `MAX_CALL_RETRIES = 2`

재시도는 두 층으로 존재한다.

- 모델 호출 재시도: JSON 파싱 실패나 예외 대응
- time stage 재시도: 다른 프레임 인덱스를 다시 고르게 하는 재질의

다만 아래에서 보겠지만, time 재시도는 "정말 틀린 사고 시점"을 검증해서 다시 묻는 구조는 아니다. 이건 이 파일의 중요한 한계다.

## 4. vLLM 로딩 구조

### 4.1 런타임 호환성 체크

`ensure_runtime_compatibility()`는 Python 3.8 이상만 확인한다.  
엄격한 의존성 검증보다는 최소한의 가드다.

### 4.2 rope validation 패치

`patch_transformers_rope_validation_compat()`  
`qwen_frame_vllm.py:100-124`

이 함수는 `transformers`와 `vLLM` 사이의 Qwen 3.5 config 호환성 충돌을 우회하기 위한 패치다.  
즉, 이 파일은 단순 추론 코드가 아니라 **Qwen3.5 + vLLM 조합을 실제로 띄우기 위한 운영형 스크립트** 성격도 갖고 있다.

### 4.3 실제 vLLM 엔진 생성

`load_vllm_model()`  
`qwen_frame_vllm.py:132-168`

중요 포인트는 다음과 같다.

- `tensor_parallel_size`를 `CUDA_VISIBLE_DEVICES` 개수로 추론
- `dtype="bfloat16"`
- `max_model_len=32768`
- `gpu_memory_utilization=0.90`
- `limit_mm_per_prompt={"image": MAX_VIDEO_SAMPLED_FRAMES}`
- `mm_processor_kwargs={"min_pixels": MIN_PIXELS, "max_pixels": MAX_PIXELS}`
- `max_num_seqs=1`

즉, 이 코드는 한 번에 많은 요청을 병렬 처리하려는 구조가 아니라, **긴 멀티이미지 입력 하나를 안정적으로 처리하는 구조**에 가깝다.

## 5. 추론 과정의 본질: "비디오"를 직접 푸는 게 아니라 "프레임 시퀀스 -> 핵심 프레임"으로 푼다

이 파일을 이해할 때 가장 중요한 포인트다.

많이들 아래처럼 생각하기 쉽다.

- time: 비디오 기반
- location: 프레임 기반
- type: 프레임 기반

하지만 실제 구현은 조금 다르다.

### 실제로는 이렇게 동작한다

1. OpenCV로 비디오를 샘플링해서 여러 장의 PIL 이미지로 만든다.
2. 그 이미지들을 시간순으로 Qwen에 넣는다.
3. Qwen은 `frame_index`를 반환한다.
4. 그 `frame_index`를 다시 `timestamp`로 바꾼다.
5. 그 시각의 **원본 비디오 프레임**을 다시 추출한다.
6. 이후 추론은 모두 그 1장으로 수행한다.

즉 time stage도 사실상 "비디오 자체 추론"이라기보다 **chronological frame sequence reasoning**이다.

이 구조의 장점은 분명하다.

- vLLM에서 멀티이미지 입력을 안정적으로 다루기 쉽다
- 각 프레임에 명시적인 타임스탬프를 붙일 수 있다
- 모델이 "몇 번째 샘플 프레임"인지 분명하게 답할 수 있다

반면 단점도 분명하다.

- 샘플 사이에 실제 첫 충돌이 있으면 놓칠 수 있다
- 위치/유형은 단일 프레임에 과하게 의존한다
- time stage가 틀리면 뒤 단계가 전부 흔들린다

---

## 6. time stage 추론 과정 상세 분석

이 파일에서 가장 중요한 부분이다.

## 6.1 샘플 프레임 생성

핵심 함수는 `load_video_frames_for_vlm()`이다.  
`qwen_frame_vllm.py:610-670`

이 함수는 다음 일을 한다.

1. OpenCV로 비디오 열기
2. `raw_fps`와 `total_frames` 읽기
3. 목표 샘플링 FPS에 맞춰 `step` 계산
4. `frame_indices` 생성
5. 마지막 프레임이 누락되지 않게 강제로 추가
6. 프레임 수가 64장을 넘으면 균등 간격으로 다시 축소
7. 해당 인덱스 프레임만 읽어서 PIL 이미지로 변환
8. 각 프레임의 실제 timestamp 생성

여기서 핵심 수식은 이 부분이다.

```python
step = max(1, int(round(raw_fps / max(sample_fps, 0.1))))
frame_indices = list(range(0, total_frames, step))
```

예를 들어 원본이 30fps이고 `VIDEO_FPS = 2.0`이면 대략 15프레임마다 한 장씩 고른다.  
즉 time stage는 실제로는 "매 프레임 판독"이 아니라 **간헐 샘플링 기반 탐색**이다.

또 하나 중요한 부분은 64장 상한이다.

```python
if len(frame_indices) > max_frames:
    ...
    frame_indices = sorted(set(reduced))
```

이건 긴 영상에서 멀티모달 토큰 수를 통제하기 위한 장치다.  
대신 긴 영상일수록 시간 해상도가 더 거칠어질 수 있다.

### time stage에서 모델이 실제로 보는 것

모델은 "동영상 파일"을 보는 게 아니라 아래 묶음을 본다.

- 자연어 프롬프트 1개
- `Frame 0 at 0.000 seconds.` 같은 텍스트 라벨
- 그 뒤의 실제 이미지
- 다시 `Frame 1 at ...` 텍스트
- 다시 이미지

즉 시간 추론은 사실상 **이미지 시퀀스 QA**에 가깝다.

## 6.2 time prompt가 모델에게 요구하는 것

`build_time_prompt()`  
`qwen_frame_vllm.py:172-262`

이 프롬프트는 다음 요소로 구성된다.

- 메타데이터
- 샘플 프레임별 timestamp catalog
- earliest frame 선택 규칙
- 출력 JSON 스키마
- 재시도 시 `failed_indices` 경고

특히 중요한 문구는 아래다.

- `Choose the earliest frame_index where the accident clearly begins.`
- `Select the first frame where physical contact begins, or where impact is clearly immediate and unavoidable.`

이 프롬프트는 "가장 이른 충돌 시작 프레임"을 고르게 하지만, 동시에  
`physical contact begins`와 `impact is clearly immediate and unavoidable`를 함께 허용한다.

이건 추론 품질 관점에서 미묘하다.

- 전자는 first-contact 기준
- 후자는 collision-just-before-contact까지 허용할 여지

즉, 코드 의도는 first-contact 탐지인데 프롬프트는 약간 더 넓은 개념을 허용하고 있다.  
시간 오차가 조금 이르게 나올 수 있는 구조적 원인이다.

## 6.3 time stage 입력을 별도로 저해상도 축소하는 이유

`call_qwen_for_frame_sequence()`  
`qwen_frame_vllm.py:823-867`

이 함수는 time stage에 들어갈 각 프레임을 다음 함수로 먼저 축소한다.

- `resize_image_to_pixel_budget()`
- `TIME_STAGE_MAX_PIXELS = MIN_PIXELS`

즉, time stage에서는 원본 해상도보다 훨씬 작은 시각 정보만 준다.

이 설계의 의미는 명확하다.

- 목적: 정밀한 충돌 부위 식별이 아니라, 사건이 "언제 시작되었는지" 찾기
- 장점: 토큰과 GPU 메모리 절약
- 단점: 작은 물체, 멀리 있는 접촉, 야간 장면에서 first-contact를 놓칠 수 있음

시간을 먼저 싸게 찾고, 이후 정밀 작업은 단일 프레임에서 하겠다는 분업이다.

## 6.4 ChatML과 멀티이미지 입력 구성

`render_chatml_prompt()`  
`qwen_frame_vllm.py:743-770`

이 함수는 message list를 Qwen/vLLM용 ChatML 문자열과 이미지 배열로 바꾼다.

이미지는 텍스트 안에서 아래 토큰으로 자리만 표시된다.

```text
<|vision_start|><|image_pad|><|vision_end|>
```

실제 이미지 데이터는 `multi_modal_data["image"]`로 따로 전달된다.  
즉 프롬프트 문자열과 픽셀 데이터가 분리된 vLLM 입력 구조다.

## 6.5 실제 모델 호출

`run_vllm_request()`  
`qwen_frame_vllm.py:779-821`

동작 순서는 단순하지만 중요하다.

1. `messages_to_vllm_request()`로 request 생성
2. `model.generate([request], sampling_params=...)` 호출
3. 첫 번째 output text 추출
4. 원문 출력 로그 저장
5. JSON 파싱 시도
6. 성공하면 dict 반환
7. 실패하면 최대 2번까지 재시도

여기서 핵심은 **모델의 자유 출력 전체를 신뢰하지 않고 JSON만 뽑아낸다**는 점이다.

## 6.6 JSON 복원 로직이 하는 일

`extract_first_json_object()`  
`qwen_frame_vllm.py:456-490`

이 함수는 생각보다 중요하다. 실제 운영 안정성의 상당 부분이 여기 달려 있다.

동작 순서는 다음과 같다.

1. `</think>` 이후 텍스트만 사용
2. `//` 주석 제거
3. `NaN`, `Infinity` 같은 값을 치환
4. 텍스트 안의 모든 `{ ... }` 후보를 스택으로 추출
5. 뒤에서부터 유효한 JSON object를 찾음
6. `frame_index`, `center_x`, `type` 같은 핵심 키가 있으면 채택

이 방식의 의미는 분명하다.

- 모델이 앞부분에 설명을 길게 써도 버틸 수 있다
- `<think>`가 있어도 마지막 JSON만 회수 가능하다
- 완벽한 instruction following이 아니어도 최대한 살린다

즉 이 파일은 프롬프트만 믿지 않고, **후처리로 모델 실수를 견디는 구조**를 갖고 있다.

## 6.7 time 결과 확정 로직

`main()`의 time loop  
`qwen_frame_vllm.py:1055-1083`

time stage는 아래처럼 굴러간다.

1. 샘플 프레임 시퀀스로 모델 호출
2. `frame_index`가 정수인지 확인
3. 범위 안인지 확인
4. 유효하면 해당 timestamp를 `accident_time`으로 채택
5. 아니면 `failed_indices`에 넣고 다시 시도

여기서 중요한 건 **검증 수준이 구조적 검증에 머문다**는 점이다.

- 정수인가
- 인덱스 범위 안인가
- 이전에 실패한 값인가

반대로 다음 검증은 없다.

- 정말 first-contact인가
- 직전 프레임에서는 미접촉인가
- 너무 이르거나 너무 늦지 않은가
- 충돌이 아니라 near-miss는 아닌가

즉, 이 파일의 time stage는 "의미 검증"보다 **형식 검증**이 강하다.

또 하나 중요한 점은 `failed_indices`의 의미다.

프롬프트에는 "이전 인덱스는 incorrect였다"고 쓰이지만, 코드상 그 인덱스가 정말 의미적으로 틀렸는지 판정하는 단계는 없다.  
실제로는 주로 아래 경우가 쌓인다.

- 범위를 벗어난 값
- 파싱 실패 후 다시 얻은 이상한 값
- 이미 제외된 값 재등장

즉 `failed_indices`는 엄밀히 말하면 "검증상 배제된 인덱스"이지, "사고 시점으로 틀렸음이 확인된 인덱스"는 아니다.

## 7. 핵심 프레임 추출 과정

time stage가 끝나면 이 파일은 선택된 샘플 프레임 이미지를 그대로 쓰지 않는다.  
대신 **원본 비디오에서 같은 시각의 실제 프레임을 다시 뽑는다.**

핵심 함수는 `extract_frame_at_time()`이다.  
`qwen_frame_vllm.py:688-741`

동작은 아래와 같다.

1. 비디오 다시 열기
2. FPS 읽기
3. FPS가 이상하면 메타데이터의 `no_frames / duration`으로 보정 시도
4. `frame_index = int(accident_time * fps)` 계산
5. 먼저 `CAP_PROP_POS_FRAMES`로 점프
6. 실패하면 `CAP_PROP_POS_MSEC`로 재시도

이 단계가 매우 중요하다.  
time stage에서 사용한 것은 **샘플 프레임의 인덱스**였고, 여기서는 **원본 비디오의 실제 프레임 인덱스**를 다시 계산한다.

즉 파이프라인은 아래 변환을 수행한다.

`샘플 프레임 인덱스 -> timestamp -> 원본 프레임 인덱스`

이게 왜 중요하냐면, location/type stage는 저해상도 샘플 프레임이 아니라 **원본 기반 프레임**을 보게 되기 때문이다.  
그래서 time stage는 거칠지만, 후속 단계는 상대적으로 더 선명한 시각 정보를 사용할 수 있다.

## 8. location stage 추론 과정

`build_location_prompt()`  
`qwen_frame_vllm.py:264-346`

여기서는 time stage와 달리 단일 이미지 1장만 입력한다.  
프롬프트는 다음을 요구한다.

- 충돌의 주된 접촉 영역을 보라
- 정규화 좌표를 내라
- 차량 중심이 아니라 실제 접촉 영역 중심을 찍어라

이후 `call_qwen_for_media(..., "image", ...)`가 실행된다.  
`qwen_frame_vllm.py:869-922`

image 모드에서는 다음 순서다.

1. 저장된 JPG를 열어 RGB PIL 이미지로 변환
2. 이미지를 먼저 content에 넣음
3. 프롬프트 텍스트를 그 뒤에 붙임
4. system message와 user message를 조합해 vLLM 호출

이 구조는 time stage와 달리 이미지가 1장뿐이라 훨씬 단순하다.

### location 결과 검증

`validate_location_prediction()`  
`qwen_frame_vllm.py:559-568`

검증은 꽤 보수적이다.

- `center_x`, `center_y`를 float로 변환
- 범위를 벗어나면 `0.0 ~ 1.0`으로 clamp

즉 좌표가 조금 이상해도 완전히 버리지 않고 살리는 구조다.

### location 추론의 의미

이 단계는 사실 "bounding box를 계산한다"기보다 모델에게 **충돌 중심점 하나를 직접 찍게 하는 방식**이다.  
그래서 구현은 단순하지만, 라벨 정의가 "관련 객체 최소 bounding box 중심"이라면 이 프롬프트와 평가 기준 사이에 차이가 생길 수 있다.

특히 이 문구가 중요하다.

- `the center of the actual contact region, not the center of the whole vehicle`

즉 현재 구현은 **접촉점 중심**에 더 가깝다.  
데이터셋 정답 정의가 bbox 중심이라면 이건 구조적 오차 원인이 될 수 있다.

## 9. type stage 추론 과정

`build_type_prompt()`  
`qwen_frame_vllm.py:348-432`

이 단계도 location과 동일하게 단일 프레임 1장으로 수행된다.  
차이는 프롬프트 안에 사고 유형 정의가 직접 포함된다는 점이다.

허용 라벨은 정확히 5개다.

- `rear-end`
- `head-on`
- `sideswipe`

이후 검증 함수 `validate_type_prediction()`  
`qwen_frame_vllm.py:570-578`

즉 이 단계의 검증도 의미 검증보다는 **폐쇄 집합 검증**이다.

## 10. 이 파일에서 "추론"이 실제로 어떻게 분해되는가

이 파일의 추론은 한 번의 큰 판단이 아니다.  
서로 다른 수준의 추론이 순차적으로 연결된다.

### 10.1 1차 추론: 거친 시간 탐색

- 입력: 저해상도 샘플 프레임 시퀀스
- 출력: `frame_index`
- 목적: 사고 시작 시점의 후보 찾기

여기서는 정밀한 충돌 부위보다 **언제 사건이 시작됐는지**가 중요하다.

### 10.2 2차 추론: 원본 프레임 재구성

- 입력: `accident_time`
- 출력: 실제 프레임 이미지
- 목적: 후속 추론에 쓸 anchor frame 확보

이 단계는 모델 추론은 아니지만, 전체 정확도에 큰 영향을 준다.  
여기서 추출 프레임이 어긋나면 이후 모든 단계가 잘못된다.

### 10.3 3차 추론: 위치 정밀화

- 입력: 단일 핵심 프레임
- 출력: `center_x`, `center_y`
- 목적: 충돌 중심점 로컬라이즈

### 10.4 4차 추론: 의미 분류

- 입력: 같은 핵심 프레임
- 출력: `type`
- 목적: 충돌 관계를 범주화

즉 이 파일은 사실상 다음 구조다.

`coarse temporal reasoning -> exact frame recovery -> spatial reasoning -> semantic classification`

이 순서가 이 파일의 본질이다.

## 12. 코드상 눈에 띄는 한계와 해석 포인트

이 파일은 잘 짜여 있지만, 추론 관점에서 몇 가지 중요한 특징과 한계가 있다.

### 12.1 시간 검증은 강하지 않다

`validate_frame_index_prediction()`은 인덱스 범위만 본다.  
즉 "맞는 사고 시점인가"는 검증하지 않는다.

결과적으로 time stage의 품질은 거의 전적으로

- 프롬프트
- 샘플링 품질
- 모델 판단

에 의존한다.

### 12.2 선언만 있고 실제로 안 쓰이는 요소가 있다

다음은 현재 코드 흐름에서 사실상 사용되지 않는다.

- `TIME_MATCH_THRESHOLD`
- `SNAP_TIME_TO_VIDEO_FPS_GRID`
- `snap_time_to_grid()`
- `strip_thinking_text()`
- `CUDA_MEMORY_RESERVE_MIB`

즉 코드가 발전하는 과정에서 실험 흔적이 남아 있는 상태로 보인다.

### 12.3 위치 프롬프트와 라벨 정의가 다를 가능성

location prompt는 "actual contact region center"를 요구한다.  
하지만 평가 기준이 "관련 객체 최소 bbox 중심"이면 이 둘은 다르다.

이 차이는 작은 프롬프트 차이처럼 보이지만, 실제 스코어에는 크게 작용할 수 있다.

### 12.4 time stage는 coarse, location/type은 single-frame이라 오차 전파가 크다

이 구조는 빠르고 실용적이지만, 한 단계가 틀리면 다음 단계가 자동으로 틀릴 가능성이 크다.

- time이 조금 이르면 아직 접촉 전 프레임을 볼 수 있다
- time이 조금 늦으면 이미 충돌 후 자세 변화만 보는 프레임이 된다
- 그러면 location은 실제 첫 충돌점이 아닌 사후 중심을 찍을 수 있다
- type도 첫 접촉 관계가 아니라 충돌 후 자세를 보고 오분류할 수 있다

즉 이 파일의 성능 병목은 결국 time stage다.

### 12.5 보안 관점에서 주의할 점

`HF_TOKEN`이 코드에 하드코딩되어 있다.  
운영 스크립트에서는 환경변수 외부 주입으로 빼는 편이 안전하다.

## 14. 추론 과정만 다시 아주 짧게 정리

1. 비디오를 2fps 수준으로 샘플링한다.
2. 최대 64장 프레임을 시간순으로 Qwen에 보여준다.
3. 모델은 첫 사고 시작 프레임의 `frame_index`를 JSON으로 반환한다.
4. 그 인덱스를 실제 `accident_time`으로 바꾼다.

7. 같은 프레임으로 `type`을 분류한다.
8. 결과와 raw log를 저장한다.

이게 바로 이 파일의 추론 구조다.

## 15. 실제 추론 프롬프트는 어떻게 구성되는가

지금까지는 파이프라인 관점에서 설명했지만, 이 파일을 제대로 이해하려면 모델이 **실제로 어떤 프롬프트를 받는지**도 봐야 한다.

`qwen_frame_vllm.py`는 단순히 자연어 한 덩어리를 보내는 구조가 아니다.

- `system` 메시지로 출력 형식을 강하게 고정하고
- `user` 메시지 안에 실제 task prompt와 이미지들을 함께 넣고
- stage마다 time/location/type에 맞는 별도 prompt를 사용한다.

즉 추론은 항상

**system prompt + stage별 user prompt + 이미지들**

의 조합으로 이루어진다.

### 15.1 공통 system prompt

핵심 코드는 아래다.

- time stage: `qwen_frame_vllm.py:850-867`
- image stage: `qwen_frame_vllm.py:904-922`

`DEBUG_VISIBLE_EVIDENCE = True`일 때 system prompt는 사실상 아래 의미를 갖는다.

- 정확히 하나의 JSON object만 반환할 것
- markdown 금지
- code fence 금지
- 답은 반드시 제공된 frame/media에 근거할 것

구체 문자열은 다음과 같다.

- time stage: `Ground every answer in the provided frames.`
- image stage: `Ground every answer in the provided media.`

즉 이 system prompt의 역할은 두 가지다.

1. 출력 포맷을 JSON으로 강제한다.
2. 모델이 일반 상식으로 추론하지 말고, **지금 입력된 시각 자료에만 근거하도록** 압박한다.

### 15.2 time stage에서 user prompt는 어떻게 들어가는가

`call_qwen_for_frame_sequence()`  
`qwen_frame_vllm.py:823-867`

이 함수에서 `user` content는 다음 순서로 만들어진다.

1. 먼저 `build_time_prompt()`가 만든 긴 텍스트 프롬프트
2. `Frame 0 at 0.000 seconds.` 같은 텍스트 라벨
3. 해당 프레임 이미지
4. 다시 다음 프레임의 텍스트 라벨
5. 다시 다음 프레임 이미지

즉 모델 입장에서는 time stage 입력이 이렇게 보인다.

- 먼저 과제 설명을 읽고
- 그 다음 시간순으로 정렬된 frame-label/image 쌍들을 본다.

여기서 중요한 건 **프롬프트가 먼저 오고, 프레임 시퀀스가 뒤따른다**는 점이다.  
그래서 모델은 먼저 "무엇을 찾아야 하는지"를 읽고 난 뒤, 실제 프레임들을 시간순으로 스캔하게 된다.

### 15.3 time prompt 본문은 어떤 구조인가

`build_time_prompt()`  
`qwen_frame_vllm.py:172-262`

이 프롬프트는 대략 5개 블록으로 구성된다.

1. 재시도 경고문
2. 역할 부여
3. 메타데이터
4. 프레임 timestamp catalog
5. 지시사항 + 출력 규칙

#### 1) 재시도 경고문

`failed_indices`가 있을 때만 들어가는 블록이다.

- 이전에 선택했던 frame indices는 incorrect였다
- 같은 인덱스를 다시 고르지 말라
- 다른 frame index를 선택하라

즉 이 문구는 모델에게 단순 재질의가 아니라 **이전 답을 배제한 재판단**을 요구한다.

#### 2) 역할 부여 문장

프롬프트 첫머리는 이렇게 시작한다.

- `You are an expert traffic accident analyst looking at a chronological sequence of sampled CCTV frames.`

이 문장의 역할은 모델을 일반 VLM이 아니라 **교통사고 분석가 역할**로 고정하는 것이다.

#### 3) task 선언

그다음 핵심 목적을 이렇게 명시한다.

- `identify the FIRST sampled frame where a traffic accident clearly begins`

여기서 중요한 단어는

- `FIRST`
- `sampled frame`
- `clearly begins`

이다.  
즉 원본 비디오의 모든 프레임이 아니라, **샘플링된 프레임 목록 안에서** 첫 사고 시작 프레임을 고르게 한다.

#### 4) 메타데이터 블록

이 블록에는 아래 정보가 들어간다.

- `region`
- `scene_layout`
- `weather`
- `day_time`
- `quality`
- `duration`
- `no_frames`
- `frame_height`
- `frame_width`

즉 모델은 시각 정보만 받는 게 아니라, 장면의 환경적 prior도 함께 받는다.

예를 들어

- 야간인지
- 날씨가 어떤지
- 화질이 어떤지
- 교차로/도로 구조가 어떤지

같은 힌트를 간접적으로 받는다.

#### 5) 프레임 catalog

모든 샘플 프레임의 index와 timestamp를 쉼표로 쭉 나열한다.

예를 들면 의미상 이런 형태다.

- `0: 0.000s, 1: 0.500s, 2: 1.000s, ...`

이 블록은 중요하다. 왜냐하면 모델이 이미지들만 보고 막연히 "중간쯤"을 고르는 게 아니라, **명시적인 index 체계와 시간축 위에서 판단**하게 만들기 때문이다.

#### 6) 지시사항 블록

핵심 규칙은 다음과 같다.

- 프레임을 순서대로 분석할 것
- 사고가 시작되는 가장 이른 frame_index를 고를 것
- physical contact가 시작되거나, impact가 즉각적이고 불가피한 첫 프레임을 고를 것
- 사고가 샘플 프레임 사이에서 시작되면, 이미 시작된 것이 분명히 보이는 가장 이른 샘플 프레임을 고를 것
- 위치와 타입은 지금 단계에서 무시할 것

즉 time prompt는 모델의 주의를

- "언제 시작됐나"
- "가장 처음인가"
- "샘플링 간격을 고려하라"

에 집중시킨다.

#### 7) 출력 규칙 블록

`DEBUG_VISIBLE_EVIDENCE = True`일 때 time stage는 아래 key를 요구한다.

- `frame_index`
- `confidence`
- `evidence`

즉 이 파일은 단순히 정답 인덱스만 받으려는 게 아니라,

- 얼마나 자신 있는지
- 무엇을 근거로 봤는지

도 함께 받으려 한다.

### 15.4 location stage 프롬프트는 어떻게 생겼는가

`build_location_prompt()`  
`qwen_frame_vllm.py:264-346`

location prompt는 time prompt와 다르게 **단일 key frame 분석**에 맞춰져 있다.

구성은 다음과 같다.

1. 역할 부여
2. 현재 이미지가 first clear accident moment라는 설명
3. 메타데이터
4. 위치 추정 과제 선언
5. 좌표 규칙 설명
6. JSON 출력 규칙

핵심 문구는 아래다.

- `looking at a single key frame from CCTV footage`
- `This image corresponds to the FIRST clear moment of a traffic accident`
- `precisely localize the primary collision point in this frame`

즉 모델은 이 단계에서

- 더 이상 시간 추론을 하지 않고
- 이미 사고 시점은 정해졌다고 가정한 채
- 그 프레임 안에서 충돌 중심만 찾게 된다.

특히 위치 prompt는 좌표 정의를 꽤 분명하게 준다.

- `center_x`: left 0.0 -> right 1.0
- `center_y`: top 0.0 -> bottom 1.0
- 차량 전체 중심이 아니라 실제 contact region의 중심을 찍을 것

즉 location prompt는 **정규화 좌표계 + 접촉 부위 중심**을 명시적으로 강제한다.

### 15.5 location stage에서 이미지와 텍스트 순서

`call_qwen_for_media()`  
`qwen_frame_vllm.py:869-922`

image 모드에서는 user content 순서가 time stage와 조금 다르다.

1. 먼저 이미지
2. 그 다음 프롬프트 텍스트

즉 location/type stage는 time stage처럼 "프롬프트 먼저, 이미지 나중"이 아니라,

- 이미지를 먼저 제시하고
- 그 이미지에 대해 무엇을 하라는 설명을 뒤에 붙인다.

이 순서 차이는 작은 구현 차이처럼 보여도, 실제로는 모델이 **이미지를 먼저 anchor로 잡은 뒤 그 이미지에 대한 과제를 읽게 만든다**는 의미가 있다.

### 15.6 type stage 프롬프트는 어떻게 생겼는가

`build_type_prompt()`  
`qwen_frame_vllm.py:348-432`

type prompt는 location prompt와 같은 single-frame 기반이지만, 그 안에 **클래스 정의가 직접 들어간다**는 점이 다르다.

구성은 아래와 같다.

1. 역할 부여
2. 이 프레임이 first clear accident moment라는 설명
3. 메타데이터
4. 5개 사고 유형 정의
5. 정확히 하나를 고르라는 지시
6. JSON 출력 규칙

핵심은 유형 정의를 프롬프트 내부에 직접 넣었다는 점이다.

- `rear-end`
- `head-on`
- `sideswipe`
- `t-bone`
- `single`

즉 모델이 사전학습 지식에 의존해서 제멋대로 분류하지 않도록, **이 스크립트 안에서 문제 정의를 다시 내려준다.**

특히 type prompt는 아래 판단 축을 사실상 유도한다.

- 차량 진행 방향 관계
- 앞/뒤/측면 접촉 면
- 다른 차량이 있었는지 여부
- 고정물 충돌인지 여부

### 15.7 stage별 프롬프트 목적 차이

세 프롬프트는 비슷해 보이지만 실제 목적은 꽤 다르다.

- time prompt: `언제 시작됐는가`
- location prompt: `어디서 부딪혔는가`
- type prompt: `어떤 기하 구조의 충돌인가`

즉 같은 이미지/영상 추론이라도, 각 stage는 서로 다른 질문을 한다.

이 분리가 중요한 이유는 모델이 한 번에 여러 가지를 동시에 판단하면 흔들릴 수 있기 때문이다.  
`qwen_frame_vllm.py`는 이를 막기 위해 stage마다 프롬프트 안에서 명시적으로

- 지금은 위치/타입 무시
- 지금은 타입 무시
- 지금은 오직 하나의 type만 고르기

처럼 **주의 대상을 좁혀 준다.**

### 15.8 이 파일에서 프롬프트 설계가 갖는 의미

결국 `qwen_frame_vllm.py`의 프롬프트 설계는 아래 철학을 따른다.

1. system prompt로 형식을 강제한다.
2. user prompt로 task를 좁게 정의한다.
3. 메타데이터로 장면 prior를 제공한다.
4. 프레임 라벨과 timestamp로 시간축을 명시한다.
5. JSON schema로 출력 후처리를 쉽게 만든다.

즉 이 파일의 추론 성능은 모델 자체만으로 결정되지 않는다.  
실제로는 **어떤 질문을 어떤 순서와 어떤 형식으로 던졌는가**가 꽤 큰 비중을 차지한다.

## 16. 프롬프트 관점에서 본 이 파일의 장점과 한계

프롬프트만 놓고 봐도 장점과 한계가 분명하다.

### 장점

- stage별 역할이 분리돼 있어서 모델이 한 번에 너무 많은 걸 판단하지 않아도 된다.
- 메타데이터와 timestamp catalog가 함께 들어가서 시간축 해석이 쉬워진다.
- JSON schema를 매우 강하게 요구해서 후처리 안정성이 높다.
- `confidence`, `evidence`를 받아 디버깅과 실패 분석이 가능하다.

### 한계

- time prompt가 여전히 `impact is clearly immediate and unavoidable`를 허용해서 first-contact보다 약간 이른 프레임을 고를 여지가 있다.
- location prompt는 contact region 중심을 요구하지만, 정답 정의가 bbox 중심이면 평가 기준과 어긋날 수 있다.
- type prompt는 single-frame 기반이라 접촉 직전/직후의 관계를 충분히 못 볼 수 있다.
- image stage에서 이미지가 먼저 오고 텍스트가 뒤에 와서, 일부 경우에는 task framing이 time stage보다 약할 수도 있다.

즉 이 파일의 프롬프트는 꽤 잘 설계되어 있지만, 특히 time과 location 정의에서는 아직 평가 기준과 완전히 일치하지 않을 가능성이 있다.
