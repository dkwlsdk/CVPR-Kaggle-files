# Zero-shot 시간 추론 가이드

## 1. 목적과 기준

이 문서는 CCTV 사고 추론에서 **시간 정확도**만 따로 떼어 집중적으로 다룬다. 목표는 현재 [`qwen_prompt.py`](/root/Desktop/workspace/ja/qwen_prompt.py)와 [`qwen_run.py`](/root/Desktop/workspace/ja/qwen_run.py) 기반 baseline을 출발점으로 삼아, zero-shot 환경에서 `accident_time`을 더 정확하게 찾기 위해 시도할 수 있는 방법론과 실험 설계를 정리하는 것이다.

이 문서에서 사용하는 시간 라벨 기준은 명확하다.

- 사고는 **물리적 충돌**이 있을 때만 성립한다.
- 사고 시간은 **첫 물리적 접촉 순간**이다.
- 충돌로 이어지지 않는 미끄러짐, 급회피, 제어력 상실은 사고가 아니다.
- 다중 차량 연쇄 충돌은 **관련된 첫 접촉 쌍의 첫 접촉 시점**을 기준으로 한다.

즉, 시간 추론은 "위험해 보이는 순간"을 찾는 문제가 아니라, **실제로 처음 닿는 프레임을 찾는 문제**다.

## 2. 현재 baseline과 주요 병목

현재 baseline은 다음 구조를 갖고 있다.

1. 비디오 전체를 Qwen에 넣어 사고 시간을 추론한다.
2. `MAX_TIME_RETRIES = 3` 범위에서 재시도한다.
3. 이전 실패값과 너무 가까운 시간은 `TIME_MATCH_THRESHOLD = 0.1`로 제외한다.
4. 유효한 시간이 나오면 그 시점 프레임을 추출해 후속 위치/타입 단계로 넘긴다.

현재 baseline의 핵심 파라미터와 구조적 특징은 아래와 같다.

| 항목 | 현재 설정 | 의미 | 시간 정확도 관점의 한계 |
|---|---|---|---|
| 샘플링 | `VIDEO_FPS = 1.0` | 비디오 입력 시 초당 1프레임 수준으로 힌트 제공 | 첫 접촉처럼 짧은 순간을 놓치기 쉽다 |
| 질의 단위 | 전체 비디오 단일 질의 | 모델이 영상 전체를 한 번에 보고 시간 하나를 반환 | 긴 영상에서 attention이 분산되고 early/late bias가 커진다 |
| 재시도 | 최대 3회 | 실패 시 다른 값을 다시 요청 | 근거 보강이 아니라 다른 답을 유도하는 수준에 머문다 |
| 검증 | 범위 검증 중심 | 0 이하, duration 초과, 지나치게 이른 값 차단 | "실제 접촉인가"를 판단하는 의미 검증은 약하다 |

현재 prompt에도 명확한 충돌 지점이 있다. [`qwen_prompt.py`](/root/Desktop/workspace/ja/qwen_prompt.py)에는 다음 문구가 포함되어 있다.

- `the first frame where physical contact begins`
- `the first frame where collision is clearly unavoidable and immediate`

첫 번째 문구는 현재 라벨 기준과 맞지만, 두 번째 문구는 그렇지 않다. "불가피한 충돌 직전"은 아직 물리적 접촉이 없는 상태일 수 있기 때문에, stricter한 first-contact 정의와 충돌한다. 시간 정확도를 높이려면 이 모호성을 먼저 제거해야 한다.

또한 실제 raw log인 [`qwen_bone_all/raw_outputs.jsonl`](/root/Desktop/workspace/ja/qwen_bone_all/raw_outputs.jsonl)을 보면 time stage에서 다음과 같은 실패 패턴이 이미 드러난다.

- 비디오의 결과 상태만 보고 사고가 있었을 것이라고 추정함
- JSON list와 설명문이 뒤섞인 출력이 나옴
- "차가 멈춰 있다" 같은 사후 상태를 근거로 시간을 고름

예를 들어 첫 번째 로그는 차량이 멈춰 있는 상태를 근거로 `12.34`초를 반환한다. 이건 첫 접촉 탐지라기보다 **사후 장면 해석**에 가깝다.

## 3. 시간 정확도를 높이는 방법론

시간 정확도 향상 방법은 다섯 그룹으로 나눠 생각하는 것이 좋다.

## 3.1 정의와 프롬프트 정렬

### First-contact only

가장 먼저 해야 할 일이다. 모델이 찾을 대상이 "사고의 시작"이 아니라 **첫 물리 접촉 프레임**임을 명확히 고정해야 한다. 이 단계가 흔들리면 나머지 개선은 전부 잡음 위에서 돌아간다.

도움이 되는 이유:
- near-miss와 실제 충돌을 분리한다.
- early prediction bias를 줄인다.
- 후속 위치/타입 단계의 기준 프레임을 안정화한다.

현재 코드 적용성:
- 매우 높음. prompt 수정만으로 바로 실험 가능하다.

비용/복잡도:
- 낮음.

### Non-collision rejection

모델이 "위험한 상황"과 "실제 충돌"을 구분하도록 유도하는 방식이다. 급정지, 스핀, 차선 이탈, 미끄러짐은 사고가 아니라는 부정 조건을 명시해야 한다.

도움이 되는 이유:
- false positive 감소
- 특히 highway, night, poor-quality 장면에서 과잉 탐지를 줄임

현재 코드 적용성:
- 매우 높음. time prompt에 부정 규칙을 추가하면 된다.

비용/복잡도:
- 낮음.

### Chain-reaction first-contact rule

연쇄 충돌이 있는 경우 가장 큰 충격이 아니라 첫 접촉을 찾게 만드는 규칙이다.

도움이 되는 이유:
- multi-car pileup에서 late prediction 감소
- first-impact anchor를 고정할 수 있다

현재 코드 적용성:
- 높음. prompt와 time verification 단계에서 별도 체크 질문을 둘 수 있다.

비용/복잡도:
- 낮음.

### Reasoning 제한

현재 prompt는 `reasoning`을 짧게 쓰게 하지만, 모델이 reasoning을 길게 붙이며 사후 장면 설명으로 흘러간다. reasoning을 허용하더라도 기준을 접촉 중심으로 제한해야 한다.

도움이 되는 이유:
- 모델의 내부 초점을 결과 상태가 아니라 접촉 순간으로 유도
- log 분석이 쉬워짐

현재 코드 적용성:
- 높음.

비용/복잡도:
- 낮음.

## 3.2 Temporal search 개선

### Higher FPS

`VIDEO_FPS = 1.0`은 첫 접촉처럼 짧은 이벤트를 찾기에 거칠다. 2fps, 4fps, 또는 더 높은 fps로 탐색 범위를 늘리면 분해능이 개선될 수 있다.

도움이 되는 이유:
- first-contact 누락 감소
- late prediction 감소

현재 코드 적용성:
- 높음. 파라미터 변경으로 가능하다.

비용/복잡도:
- 중간. 입력 길이와 추론 비용이 증가한다.

### Coarse-to-fine search

전체 영상을 거칠게 보고 후보 구간을 찾은 뒤, 그 구간만 더 촘촘하게 재탐색하는 방식이다.

도움이 되는 이유:
- 긴 영상 전체 비용을 크게 늘리지 않으면서 시간 분해능을 높일 수 있다.
- short clip처럼 세밀하게 보고 싶은 장면만 다시 볼 수 있다.

현재 코드 적용성:
- 중간. 전체 파이프라인 구조를 time stage만큼은 두 단계로 나눠야 한다.

비용/복잡도:
- 중간.

### Sliding windows

전체 비디오를 한 번에 질의하지 않고, 일정 길이의 clip을 sliding window로 잘라서 각각 평가하는 방식이다.

도움이 되는 이유:
- 긴 영상에서 attention dilution 감소
- 후보 시점 주변 맥락을 더 일관되게 볼 수 있음

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간 이상.

### Clip proposal + reranking

시간 후보를 여러 개 만든 뒤, 후보별로 "정말 첫 접촉인가"를 다시 검증하는 방식이다.

도움이 되는 이유:
- 한 번의 single-shot 예측보다 안정적
- early/late bias correction에 유리

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간.

### Local refinement around candidate time

한 번 time 후보가 나오면 그 주변의 더 촘촘한 frame/clip만 다시 보는 방식이다.

도움이 되는 이유:
- coarse prediction을 fine prediction으로 바꿔준다.
- first contact와 post-impact reaction을 분리하기 쉽다.

현재 코드 적용성:
- 중간.

비용/복잡도:
- 낮음~중간.

## 3.3 검증과 재판단

### Self-consistency

같은 비디오를 조금 다른 샘플링이나 phrasing으로 여러 번 보고, 서로 비슷한 first-contact를 가리키는지 확인하는 방식이다.

도움이 되는 이유:
- 불안정한 단일 예측을 걸러낼 수 있다.

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간.

### Multi-pass voting

여러 후보 시간 중 가장 자주 지지받는 시간대를 택하는 방식이다.

도움이 되는 이유:
- outlier prediction 완화
- retry를 더 의미 있는 구조로 바꿀 수 있음

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간.

### Candidate verification

후보 시간 하나를 바로 채택하지 않고, "이 시점 직전에는 미접촉이고 이 시점에는 접촉인가"를 별도 검증 프롬프트로 다시 묻는 단계다.

도움이 되는 이유:
- late/early error를 직접 줄인다.
- current retry보다 훨씬 목적이 뚜렷하다.

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간.

### Near-miss rejection

사고처럼 보이는 위험 장면을 별도 오탐 범주로 보고 거르는 것이다.

도움이 되는 이유:
- 특히 intersection, highway 고속 장면에서 오탐 감소

현재 코드 적용성:
- 높음.

비용/복잡도:
- 낮음~중간.

### Early/late bias correction

실험 결과가 항상 조금 빠르거나 늦게 나오는지 분석하고, 그 편향을 줄이는 규칙을 세운다.

도움이 되는 이유:
- systematic bias를 빠르게 줄일 수 있다.

현재 코드 적용성:
- 높음.

비용/복잡도:
- 낮음.

## 3.4 보조 신호와 hybrid 접근

### Optical flow

프레임 간 움직임 변화량을 통해 접촉 직전과 직후의 급격한 motion pattern을 포착한다.

도움이 되는 이유:
- first contact 전후의 동적 변화를 잘 잡는다.
- poor-quality 장면에서 정적 모양보다 강할 수 있다.

현재 코드 적용성:
- 낮음~중간. 별도 전처리 또는 보조 모듈이 필요하다.

비용/복잡도:
- 중간.

### Object tracking

차량 궤적을 따라가며 거리, 겹침, 방향 관계를 추적하는 방식이다.

도움이 되는 이유:
- 실제 접촉 후보를 더 명시적으로 잡을 수 있다.
- chain reaction에서 첫 접촉 쌍을 찾는 데 유리하다.

현재 코드 적용성:
- 중간.

비용/복잡도:
- 중간~높음.

### TTC / approach speed

Time-to-collision 유사 지표나 접근 속도를 써서 접촉 임박 구간을 찾는다.

도움이 되는 이유:
- temporal proposal 생성에 강하다.
- full-video 질문보다 candidate search에 적합하다.

현재 코드 적용성:
- 낮음~중간.

비용/복잡도:
- 중간.

### Motion change detector

충돌 직후의 급격한 자세 변화, 회전, 감속을 감지하는 보조 모델이나 규칙 기반 탐지기다.

도움이 되는 이유:
- 후보 구간 탐색 속도를 높일 수 있다.

현재 코드 적용성:
- 낮음~중간.

비용/복잡도:
- 중간.

### Collision proposal model + VLM rerank

가벼운 detector가 후보 구간을 제안하고, VLM이 first-contact 여부를 다시 판단하는 구조다.

도움이 되는 이유:
- Qwen이 전체 비디오를 다 떠안지 않아도 된다.
- hybrid 구조로 역할 분담이 가능하다.

현재 코드 적용성:
- 낮음.

비용/복잡도:
- 높음.

## 3.5 더 넓은 조사 항목

### Larger VLM

더 큰 모델로 바꾸면 장면 해석은 좋아질 수 있다. 다만 시간 분해능 문제 자체가 해결되는 것은 아니다.

### Video-native model

프레임 나열이 아니라 temporal modeling 자체가 더 강한 video-native architecture를 쓰는 접근이다.

### Ensemble

여러 모델 또는 여러 temporal strategy의 결과를 합치는 방식이다.

### Zero-shot detector + language model composition

감지기와 언어 모델을 조합해, 후보 탐색과 최종 판단을 분리하는 구조다.

이 네 가지는 효과 가능성은 있지만, 현재 baseline에서 바로 붙이기보다는 **조사형 고비용 방법**으로 분류하는 편이 맞다.

## 5. 실험 설계

실험은 반드시 baseline 대비 **한 변수만 바꿔서** 진행한다. baseline은 현재 구조 그대로 둔다.

- E0: `1fps + 전체 비디오 단일 질의 + 최대 3회 재시도`

각 실험은 다음 항목을 기록한다.

- 변경점
- 가설
- 기대 효과
- 주요 failure mode
- latency / cost 증가 여부
- 적용 난이도

아래 실험 매트릭스를 기본 설계로 사용한다.

| ID | 변경점 | 가설 | 기대 효과 | 주요 failure mode | 비용 증가 | 난이도 |
|---|---|---|---|---|---|---|
| E0 | current baseline | 기준점 확보 | 비교 기준 생성 | early/late 모두 큼 | 기준 | 낮음 |
| E1 | stricter first-contact prompt | 정의 충돌 제거만으로도 정확도 개선 | non-collision 오탐 감소 | still coarse at 1fps | 낮음 | 낮음 |
| E2 | fps sweep: `1.0`, `2.0`, `4.0` | 더 높은 fps가 첫 접촉 포착에 도움 | late prediction 감소 | 비용 상승, 긴 영상 부담 | 중간 | 낮음 |
| E3 | sliding windows vs full-video | 짧은 clip이 긴 영상보다 안정적 | attention dilution 감소 | 후보 merge가 어려울 수 있음 | 중간 | 중간 |
| E4 | candidate reranking / local refinement | coarse 후보를 fine하게 보정 가능 | first-contact 근접도 향상 | 잘못된 후보에서 refinement | 중간 | 중간 |

| E7 | multi-pass voting | 단일 추론 변동성 완화 | robust time prediction | 반복적 동일 오답 | 중간 | 중간 |
| E8 | optical flow / tracking 보조 신호 | 동적 단서가 애매한 접촉을 더 잘 잡음 | hard case 개선 | 추적 실패 전파 | 높음 | 높음 |

## 6. 평가 보고와 분석 관점

이 문서는 점수 계산식 자체는 다루지 않는다. 대신 time accuracy 실험에서는 **time score와 failure analysis**만 집중해서 본다.

함께 기록할 보조 지표는 아래 네 가지로 제한한다.

- early prediction rate
- late prediction rate
- no-collision false positive count
- chain-reaction misanchor count

subset 분석은 [`CVPR-Kaggle-files/데이터셋/test_metadata.csv`](/root/Desktop/workspace/ja/CVPR-Kaggle-files/%EB%8D%B0%EC%9D%B4%ED%84%B0%EC%85%8B/test_metadata.csv)의 메타데이터 컬럼을 기준으로 나눈다.

- `weather`
- `day_time`

특히 아래 비교는 꼭 넣는 것이 좋다.

- `highway` vs `signalized_intersection` vs `city_street`
- `day` vs `night`
- `Good/Fine` vs `Poor/Very_Poor`
- 5~10초 영상 vs 25~30초 영상

## 7. 대표 실패 시나리오

시간 추론 실험에서는 아래 시나리오를 별도 failure bucket으로 관리하는 것이 좋다.

- 접촉 없이 미끄러지거나 제어를 잃는 장면
- 급회피나 급정지로 사고처럼 보이지만 실제 접촉은 없는 장면
- 첫 접촉은 빠르지만 큰 충격 반응은 늦게 나오는 rear-end
- 교차로에서 첫 접촉이 가려지는 t-bone
- 다중 차량 적체에서 첫 번째 접촉과 후속 접촉이 분리되는 장면
- `Very_Poor` 또는 `Poor` 품질의 야간 영상
- 짧은 영상과 긴 영상 사이의 편향 차이

이 버킷은 단순 정성 메모가 아니라, 각 실험에서 **어떤 실패가 줄고 어떤 실패가 그대로 남는지**를 보기 위한 기준점이다.

## 9. 결론

현재 baseline에서 시간 정확도를 가장 크게 깎는 문제는 세 가지다.

- first-contact 정의와 prompt가 완전히 정렬되어 있지 않다.
- `1fps + 전체 비디오 단일 질의` 구조가 너무 거칠다.
- retry는 있지만 실제 검증과 재판단 구조는 약하다.

따라서 가장 현실적인 개선 순서는 다음과 같다.

1. **정의와 prompt를 first-contact 기준으로 정렬한다.**
2. **fps와 탐색 구조를 바꿔 temporal resolution을 높인다.**
3. **candidate verification과 reranking으로 early/late bias를 줄인다.**
4. **필요할 때만 optical flow, tracking, hybrid 구조로 확장한다.**

이 순서로 가면, 현재 Qwen zero-shot baseline 안에서도 시간 정확도를 꽤 체계적으로 끌어올릴 수 있다.
